# 开发管理手册

## 协作模式

```
[你] → 审阅计划 → 把任务给我 → 我在终端给 Codex CLI 指令 → Codex 写代码 → [你] 审核代码
```

### 角色分工

| 角色 | 职责 |
|------|------|
| **你（Claude/监督者）** | 审阅每阶段代码、给出修改意见、批准进入下一阶段、记录审核结果 |
| **我（项目管理者）** | 拆分任务、给出命令、协调进度 |
| **Codex CLI（执行者）** | 在终端执行命令、写代码、运行测试 |

## 开发流程

### 阶段制开发

每个 Task（参见实施计划）为一个开发阶段。每个阶段经历：

```
1. 我给命令 → 2. 你用 Codex CLI 执行 → 3. Codex 写完 → 4. 我审核 → 5. 通过/返工
```

### 每个阶段的 Handoff 格式

我会用以下格式给出每个阶段的指令：

```
### Phase N, Task N: [任务名]

**目标：** [一句话]

**给 Codex CLI 的指令：**
[可以直接复制粘贴给 Codex 的命令或 prompt]

**验收标准：**
- [ ] 标准 1
- [ ] 标准 2
```

### 审核流程

每个阶段完成后，你/Codex 需要提供：
1. `git diff` 输出（改了什么）
2. 测试运行结果（`pytest` 输出）
3. 如果有 API 调用，记录 token 消耗和成本

我会审查：
- 代码是否匹配设计文档
- 测试是否充分
- 是否有安全/性能问题
- 是否符合 MVP 范围（不做过度设计）

审查结果格式：
```
### Review: Phase N, Task N
- 通过项：...
- 问题项：...
- 修改指令：...
- 判定：✅ 通过 / ⚠️ 修改后通过 / ❌ 重做
```

## Codex CLI 使用提示

### 启动 Codex CLI

在 `book-skill-generator` 目录下：
```bash
codex
```

### 给 Codex 的 Prompt 写法

Codex CLI 可以读写文件、运行命令。给它的 prompt 应该：

1. **明确当前工作目录**：`你现在在 book-skill-generator 项目中`
2. **说明依赖已安装**：`虚拟环境 venv 已经配置好，依赖已安装`
3. **给出具体文件路径**：使用相对于项目根目录的路径
4. **指明需要创建还是修改**：`创建 app/parsers/pdf_parser.py`
5. **给出验收条件**：`完成后运行 python -m pytest tests/test_pdf_parser.py -v，预期全部通过`

### 示例 Handoff

```
你现在在 book-skill-generator 项目中。虚拟环境 venv 已配置。

### 任务：创建 PDF 解析器

1. 创建文件 app/parsers/pdf_parser.py，内容是：
[完整代码]

2. 创建文件 tests/test_pdf_parser.py：
[完整测试代码]

3. 生成测试 fixtures：
python3 -c "[脚本]"

4. 运行测试：
python -m pytest tests/test_pdf_parser.py -v

预期：所有测试通过。
```

## 关键规则

### 必须
- 每个 Task 完成后必须运行测试并展示输出
- 每次 commit message 遵循 `feat:` / `fix:` / `test:` 前缀规范
- API key 永远不会出现在代码或 commit 中（只存在于 `.env`）
- 每个 Task 一个 commit，不要多个 Task 合在一个 commit

### 禁止
- 不要跳过测试直接写实现
- 不要在代码中硬编码绝对路径
- 不要引入计划之外的依赖
- 不要碰 `.env` 文件（只读 `.env.example`）
- 不要做"顺手重构"，聚焦当前 Task

### 遇到问题
- 测试跑不过 → 先看错误信息，不要盲目改
- API 调用失败 → 检查 `.env` 中的 key 和网络
- 不确定怎么实现 → 先停下，让我介入分析

## 审核检查清单

我审核每个 Task 时会检查：

- [ ] 代码是否与设计文档一致
- [ ] 测试是否覆盖了正常路径和边界情况
- [ ] 错误处理是否合理（不吞异常、不假报成功）
- [ ] 是否有安全风险（命令注入、路径遍历、密钥泄露）
- [ ] 文件组织是否合理（是否放在正确的目录）
- [ ] Commit message 是否规范
- [ ] 是否引入了不必要的依赖
