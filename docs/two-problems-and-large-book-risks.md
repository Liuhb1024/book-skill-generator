# LLM 文本退化：两次实战诊断、三层防御体系 与 大书蒸馏风险前瞻

> 基于《刻意练习》蒸馏实战，结合 Codex 独立搜索研究结论。
> 2026-06-11

---

## 一、序幕：两次退化，两种根因

2026 年 6 月 11 日，《刻意练习》EPUB 蒸馏连续出现两类文本退化。表面相似（都是输出不可读），但根因完全不同。

### 问题 1：曝光偏差 → 字符级重复循环

**现象**：

```
ch01（有目的的练习）："史蒂夫·夫夫夫夫夫夫夫夫夫夫夫夫夫夫夫夫夫夫夫夫夫..."
  → 4218 字符，仅 5 行有意义内容

ch04（黄金标准）："它要求你你你你你你你你你你你你你你你你你你你你你..."
  → 4112 字符，仅 3 行有意义内容
```

**根因分析**：

这是经典的**曝光偏差（Exposure Bias）**。LLM 训练时使用 Teacher Forcing——每一步都看到人类写的"正确答案"作为上文。但推理时，模型看到的是自己刚生成的 token：

```
训练时：  人类写的上文 → 预测下一个 token ✓
推理时：  自己生成的上文 → 预测下一个 token → 可能漂移
```

具体到"夫夫夫夫"的形成机制：假设在某一步，"夫"的概率是 8%。temperature=1.0 下严格按原始分布采样，"夫"被选中。它进入上文后，下一步"夫"在"夫"后面的 bigram 概率抬升到 15% → 25% → 40% → 锁定。这是一个正反馈循环——不是模型故障，是自回归生成在无约束时的**必然数学行为**。

**为什么 temperature=1.0 特别危险**：temperature 控制采样的"锐度"。1.0 意味着严格按原始概率分布采样，零安全边际。一旦某个 token 的重复概率开始上升，模型完全没有内在机制对抗。

**为什么 Claude 不这样但 DeepSeek 会这样**：这是 Codex 通过搜索 the-knowledge-guy 和 book-to-skill 项目源码后确认的关键结论。Claude 的 RLHF 训练已把抗重复行为**内化到模型权重**——训练过程中见过大量重复文本的负面反馈，学会了自我规避。DeepSeek 定位更偏 raw capability，把抗重复的控制权交给开发者——这是一层需要你自己搭建的安全机制。

**对比优秀项目**：the-knowledge-guy 和 book-to-skill 本质上是 Claude Code skill 编排器，章节蒸馏由 Claude Code 子代理按 Markdown 模板生成，不是一个小模型 API 的一次性长文本输出。所以在它们的代码库中，你看不到 `frequency_penalty`、`presence_penalty` 这类显式抗退化参数，也看不到 n-gram 自动检测。它们依赖的是 Claude 本身、章节切片、模板质量条款和人工可见文件来控制质量。这不是谁更先进的问题，而是架构差异决定的。

### 问题 2：分布塌缩 → 语义退化

**现象**（修复问题 1 之后出现）：

```
"天真的重复不会带带来进步。 2. 每次练必须有一个具体的目标标标标标标标标标标
才才才才才能在才能能能能能才能在才能才能才这些工作作作。 3. 只要你撑你入舒适区
区区就是没没没有你你你时你要时要时时要时时要要要在要在要在要在看在该在该在这个
其实其实其其实这里有这里有你现在在这些前这些前这些才华这些才华这些才华适合。"
```

**根因分析**：

这不是曝光偏差。这是**分布塌缩（Distribution Collapse）**——模型的 token 概率分布整体崩塌了。

区别在于：
- 曝光偏差：一个 token 自我强化，表现为**同一个字符连续不断**
- 分布塌缩：概率质量集中在极少数候选 token 上，表现为**几个 token 间交替重复，最终彻底语无伦次**

频率惩罚没有拦住的原因是：penalty 在采样前压制已出现 token 的概率，但如果概率分布本身已经塌缩到只有 3-5 个 token 有意义，压掉一个它就跳去另一个，同样在塌缩范围内。"标"被压→跳"才"→"才"被压→跳"能"→"能"被压→跳回"标"，如此循环。

**为什么是小模型特有问题**：deepseek-v4-flash 是蒸馏压缩过的，参数少。短文本看不出差距，但章节蒸馏需要输出 2000-4000 字的结构化长文本，小模型跑到后半段，内部表征开始崩塌，概率分布的"尾巴"（低概率 token）不可靠，质量集中在少数 token 上。

## 二、三层防御体系：完整方案

### 第 1 层：采样参数——四元协同

这是 API 调用层面最核心的防线。四个参数各司其职：

| 参数 | 作用 | 我们的配置（Pro 模型，章节蒸馏） |
|------|------|------|
| `temperature` | 控制分布的"锐度" | 1.0→0.8→0.6（渐进收紧） |
| `top_p` | 截断不可靠的概率尾巴 | 0.9→0.95→0.95 |
| `frequency_penalty` | token 出现次数越多，概率越低 | 0.5→0.7→1.0 |
| `presence_penalty` | token 出现过就降低概率 | 0.3→0.4→0.5 |

**top_p 是关键缺失**。DeepSeek 官方文档明确支持此参数（来源：api-docs.deepseek.com）。Nucleus Sampling 论文（Holtzman et al., ICLR 2020）证明：截断低概率尾部可缓解神经文本退化。frequency_penalty 是"用过的词少用"，top_p 是"不可靠的词不用"。两者互补。

**渐进策略的设计逻辑**：不是无脑重试，而是每次重试都更保守——降低 temperature（减少随机性）+ 收紧 top_p（切掉更多概率尾巴）+ 加大 penalty（更强力压制重复）。三次尝试覆盖从宽松到严格的全谱。

### 第 2 层：n-gram 确定检测

第 1 层是概率性的，不保证 100%。第 2 层是确定性的安全网：

- 字符级：同一字符连续 ≥ 7 次 → 触发重试（阈值从 10 降到 6，Codex 实际设为 6）
- 2-gram：同一对字符连续 ≥ 3 次 → 触发
- 3-gram：同一组字符连续 ≥ 2 次 → 触发

Codex 额外规避了 Markdown 表格分隔线（`|---|---|`）被 2-gram 检测误杀的问题。

### 第 3 层：中文语义退化检测

Codex 自创的 `_has_degenerate_cjk_window`——用滑动窗口检测中文文本的字符多样性。如果某个窗口内唯一字符占比过低，说明语义已经退化。这是比纯 n-gram 更聪明的检测，能发现"句子不通但字符不完全重复"的隐藏退化。

### 配套决策：模型升级

Codex 在搜索研究后做出的核心判断：全量切换到 deepseek-v4-pro-guan。flash 模型即使加了 top_p 也不够可靠，因为模型容量本身是瓶颈。同时将 MAX_CONCURRENT_CHAPTERS 从 10 降到 4，避免 pro 模型的限频。

**成本影响**：pro 模型输入 ¥3.0/M + 输出 ¥6.0/M，是 flash 的约 3 倍。但《刻意练习》实际蒸馏只用了 **¥0.23**，耗时 **190s**，比之前的 flash 版本还快（因为不需要反复重试）。

## 三、大书蒸馏：风险前瞻

以两本典型大书为例：

| 维度 | 史记 | 毛选（第二版） |
|------|------|---------------|
| 篇/章数 | 130 篇 | 159 篇 |
| 总字数 | ~52 万（原文）/ 160 万+（含注释译文） | ~107 万 |
| 结构复杂度 | 极高（本纪/表/书/世家/列传五体） | 高（按时期分卷，每卷内按时序排列） |
| 语言风格 | 文言文 | 现代白话（政论/演讲/书信） |
| 内容类型差异 | 叙事、表、典制、传记 四类完全不同的文本 | 论说文、报告、演讲、书信 |

### 风险 1：context window 溢出（高概率，高影响）

当前 Stage 3 整合阶段将所有章节内容拼接后传给 AI。130 章 × 每章平均 2000 字（蒸馏后） = 260,000 字 = ~350,000 tokens。即使只传章节摘要，也可能超出 DeepSeek 的 context window。

**应对方向**：Stage 3 改为分层整合——先按体例/分卷分别整合（本纪 12 章整合一次、世家 30 章整合一次、列传 70 章整合一次...），再对各分卷的整合结果做最终整合。这正是 ToM（Tree-oriented MapReduce, EMNLP 2025）论文的思路：利用书籍的自然层级结构而非任意切分。

### 风险 2：章节检测失效（中等概率，高影响）

当前 chapter_detector 用正则匹配"第X章"、"Chapter N"等模式。史记的结构完全不同：
- 没有"第X章"，而是"五帝本纪第一"、"秦始皇本纪第六"
- "表"是表格数据，不是叙述文本
- "书"是典章制度专史，结构特殊

**应对方向**：增加对"纪传体"结构（X纪第Y、X世家第Y、X列传第Y）的识别，对"表"类标记为 `bm`（back matter 或特殊处理），对无法蒸馏的内容（纯表格）跳过。

### 风险 3：成本爆炸（中等概率，高影响）

当前《刻意练习》（9 章）成本 ¥0.23。线性推算：
- 史记 130 章 ≈ ¥3.3
- 毛选 159 篇 ≈ ¥4.0

但实际可能更高，因为大书的章节平均更长（输入 token 更多）。如果单书成本突破 ¥5，闲鱼卖 ¥9.9 的利润就很薄了。

**应对方向**：在 Stage 2 之前加入**章节预筛选**——对特别短（< 500 字）或信息密度极低（如表）的章节跳过蒸馏，直接生成一个标注。在 Stage 3 之前加入**成本估算**，超预算时用 flash 模型降级处理非核心章节。

### 风险 4：章节长度极端差异（中等概率，中影响）

史记中，"五帝本纪"可能几千字，而"司马相如列传"（含大量辞赋原文）可能数万字。当前 head 8000 + tail 8000 的截断策略对后者不够——中间的被省略部分可能包含关键论证。

毛选中，"论持久战"约 5 万字，而一些短篇讲话可能只有几百字。当前策略对前者信息损失大，对后者浪费 token。

**应对方向**：根据章节长度动态调整截断策略。短章节全文传入，中等章节用 head+tail，超长章节用**三段式截断**（head + 随机中间采样 + tail）。或者采用滑动窗口摘要法。

### 风险 5：文言文理解能力（中等概率，中影响）

史记是西汉文言文，DeepSeek 对现代中文的训练数据远多于文言文。AI 可能：
- 把"走"理解为"走路"而非"跑"
- 误解古汉语虚词（之、乎、者、也）
- 对人名、地名、官名的实体识别错误

**应对方向**：在 prompt 中增加"本书为文言文/古典汉语"的提示，要求 AI 保持警惕。对比测试：用同一段文言文分别让 GPT-4o 和 DeepSeek 蒸馏，评估差距。

### 风险 6：API 内容审查（低概率，高影响）

毛选涉及政治内容。DMXAPI 作为国内代理，可能有内容审查机制。如果 AI 在蒸馏过程中遇到敏感段落被拒绝输出，整章会标记为"蒸馏失败"。

**应对方向**：在 prompt 中明确"这是学术知识蒸馏任务"，降低触发审查的概率。如果某章反复失败，检查 API 返回的错误类型（是内容审查还是网络错误）。

### 风险 7：蒸馏耗时过长（高概率，中影响）

当前 MAX_CONCURRENT_CHAPTERS=4。130 章需要约 33 轮，每轮约 60-90 秒，总耗时约 33-50 分钟。SSE 连接可能在某些网络环境下超时。而且前端用户要干等 40 分钟。

**应对方向**：
- 改为异步任务模式（提交蒸馏任务 → 后台执行 → 完成后通知下载），而非实时 SSE
- 分卷蒸馏：用户可以按"本纪/世家/列传"分批蒸馏
- 增加断点续蒸馏：失败的章节单独重试，不需要从头跑

### 风险 8：Stage 3 整合质量下降（高概率，高影响）

即使 context window 不溢出，130 章的摘要一次性塞给 AI，"lost in the middle" 效应会导致中间章节的信息被忽略。AI 倾向于更关注开头和结尾的内容。

**应对方向**：参考 LLM×MapReduce（ACL 2025）的结构化信息协议——在整合同类章节时，强制 AI 先列出所有章节的共性模式，再提取差异化洞察。概念冲突检测：如果第 3 章和第 78 章的说法矛盾，需要标记而不是强行统一。

## 四、总结

文本退化问题的本质是：你在用一个 API 参数调优的方式，手工搭建 Claude 已经通过 RLHF 内化到模型权重的安全层。这不是野路子，这是 DeepSeek 这类 raw-capability 模型的正确用法。OpenAI 的 API 同样暴露这些参数，生产环境的最佳实践中 frequency_penalty=0.5, presence_penalty=0.5, top_p=0.9 是长文本生成的标准配置。

对比优秀项目的关键认知是：**架构选择决定了问题空间。** the-knowledge-guy 选用 Claude API（模型内化安全层）+ 子代理分步生成（每步输出短，不易退化），我们选用 DeepSeek（低成本、高性能、需手动调参）+ 一键生成长文本（效率高、需防御退化）。两条路各有优劣，我们的选择带来了成本和速度优势，代价是需要自己搭建这层防御。

对于大书蒸馏，核心矛盾是：**体量增长不是线性的**。从 9 章到 130 章，不只是成本和时间翻倍的问题——context window、章节结构多样性、内容类型差异、用户等待耐心都会成为新瓶颈。当前架构在10章以内表现良好，但要处理百章级别的大书，需要在 Stage 3 引入分层整合、在 Stage 2 引入动态截断、在前端引入异步任务模式。
