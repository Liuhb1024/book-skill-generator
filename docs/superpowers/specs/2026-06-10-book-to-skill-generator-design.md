# 书籍蒸馏 Skill 生成器 — 完整设计手册

## 一、产品定义

### 1.1 做什么

一个 Web 工具，用户上传 PDF/EPUB/TXT 电子书，系统自动蒸馏成 Claude Code Skill 文件包，用户下载后在闲鱼等平台出售。

### 1.2 产品定位

**"上传一本书，10 分钟出一个 Skill"** — 全自动、低成本、零人工介入。

### 1.3 目标用户

- 在闲鱼/小红书卖 AI Skill 虚拟资料的副业者
- 想把自己读过的好书快速变成 AI 助手的人

### 1.4 商业模式

- 工具本身：自用或小范围 SaaS
- 产出物：Skill 文件包，闲鱼定价 ¥9.9-199
- 成本：单本书 API 费用 ¥0.5-2（DeepSeek），服务端几乎零成本

## 二、技术方案

### 2.1 方案选择

采用 **精简版 Map-Reduce + 两层输出架构**。

**参考来源：**

| 来源 | 借鉴内容 |
|------|---------|
| [the-knowledge-guy](https://github.com/vitalysim/the-knowledge-guy) | 两层输出架构、骨架先行、章节按原书编号、成本控制思路 |
| [book-to-skill](https://github.com/virgiliojr94/book-to-skill) (2700+ stars) | 多格式解析器链、每章 1 文件粒度、密度优先原则 |
| [cangjie-skill](https://github.com/kangarooking/cangjie-skill) | 多维度提取（框架/原则/案例/反例/术语）、去重验证思路 |

**有意识放弃的：**

| 来源 | 放弃内容 | 原因 |
|------|---------|------|
| spinedigest | 知识图谱 + 对抗式 Agent | DeepSeek 撑不住复杂推理；引入图数据库太重 |
| the-knowledge-guy | 跨书检索、互动课程、HTML artifact | 卖单本书场景不需要 |
| cangjie-skill | RIA-TV++ 三重验证、压力测试 | 依赖 Claude 级推理；DeepSeek 做不了 |

### 2.2 为什么不用 RAG

RAG 需要向量数据库（Chroma/Milvus）+ embedding 模型 + 检索管线，维护成本高。而且 RAG 擅长的是"从书中找一段话回答你"，不是"理解全书并生成结构化的方法论指南"。蒸馏场景要做的是后者——理解、整合、重构，这正是 Map-Reduce 擅长的。

### 2.3 蒸馏管线（核心引擎）

```
阶段 0：文本提取（Python，不调 AI）
  → 解析 PDF/EPUB/TXT
  → 自动识别章节目录
  → 产出 raw/ 目录（每章一个 txt + 元数据）

阶段 1：骨架提取（1 次 AI 调用）
  → 输入：目录 + 前言/导论 + 每章首尾各 1000 字
  → 输出：核心论点 + 6-10 个关键框架 + 章节主题索引

阶段 2：并行章节蒸馏（N 次 AI 调用，可并行）
  → 每章独立处理
  → 提取：核心框架、关键方法论、书中案例、反模式/误区、可执行步骤
  → 所有章节并行，速度最快

阶段 3：知识整合（1 次 AI 调用）
  → 去重：同一概念在多章出现 → 合并
  → 补漏：跨章节逻辑链是否断裂
  → 排序：按重要性/依赖关系重排

阶段 4：打包输出（Python）
  → 生成 SKILL.md + chapters/ 目录 + 使用说明
  → 打包 zip 供下载
```

#### 各阶段 Prompt 设计要点

**阶段 1 Prompt 结构：**
```
你是书籍分析专家。请根据以下信息提取本书骨架：

[书目录]
[前言/导论 2000 字]
[每章首 1000 字 + 尾 1000 字]

请输出：
1. 核心论点（100 字内）
2. 6-10 个关键框架（每个框架：名称 + 一句话描述 + 首次出现章节）
3. 章节主题索引（每章一句话概括）
4. 关键术语表（术语 + 简短定义 + 所在章节）

输出 JSON 格式。
```

**阶段 2 Prompt 结构（每章并行）：**
```
你是书籍分析专家。请分析以下章节：

章节：第 X 章《标题》
[章节全文或首尾截断]

请提取：
1. 本章核心框架（如有）
2. 关键方法论/技术（可操作、可复现的描述）
3. 书中案例（情境 + 做法 + 结果）
4. 反模式/常见误区
5. 可执行步骤清单

要求：密度优先，不要抄原文，用"实践者口吻"写。
输出 JSON 格式。
```

**阶段 3 Prompt 结构：**
```
你是知识整合专家。请整合以下内容：

[阶段 1 骨架]
[阶段 2 所有章节结果]

请做：
1. 合并跨章节的重复概念，保留最完整版本
2. 标记跨章节的逻辑链（如"第3章的X是第7章Y的理论基础"）
3. 按重要性排序框架和术语
4. 如有信息缺口，标注出来

最后生成完整的 SKILL.md 内容。
```

### 2.4 输出结构

```
《书名》Skill/
├── SKILL.md              # 常驻文件，≤4000 tokens
├── chapters/
│   ├── ch01-章节名.md
│   ├── ch02-章节名.md
│   └── ...
├── guides/
│   ├── cheatsheet.md     # 速查表（可选）
│   └── glossary.md       # 术语表（可选）
└── README.md             # 安装说明 + 使用场景 + 示例提问
```

### 2.5 SKILL.md 模板

```markdown
---
name: 《书名》
description: 基于[书名]蒸馏的 AI 技能，可扮演作者思维、提供方法论指导、快速查阅书中精华

---

# 本书核心
[100 字内概括全书核心论点]

# 关键框架
1. **框架A** — 一句话描述（详见 ch03-xxx.md）
2. **框架B** — 一句话描述（详见 ch07-xxx.md）
...

# 章节索引

|------|------|---------|
| Ch01 标题 | 一句话 | 概念1、概念2 |
...

# 触发场景
- 当用户问 X 时 → 查阅第 Y 章
- 当用户想学 Z 时 → 使用框架 A
...

# 术语速查
| 术语 | 定义 | 详见 |
|------|------|------|
...
```

## 三、降价策略（控制 API 成本）

| 策略 | 说明 | 节省幅度 |
|------|------|---------|
| 文本截断 | 阶段 2 每章只发首 8000 + 尾 8000 字 | ~40% |
| 并行调用 | 阶段 2 所有章节同时调 API | 时间节省 N 倍 |
| 模型分层 | 阶段 1/3 用 DeepSeek-V3，阶段 2 用 DeepSeek-V3-Lite | ~30% |
| 可选跳过 | 用户可不生成术语表/速查表 | ~15% |
| 缓存复用 | 同书重复处理跳过阶段 0 | 文本提取免费 |

### 成本估算（DeepSeek 定价）

| 书规模 | 阶段 1 | 阶段 2（N 章） | 阶段 3 | 合计 |
|--------|--------|---------------|--------|------|
| 200 页/10 章 | ¥0.03 | ¥0.30 | ¥0.05 | **¥0.38** |
| 400 页/20 章 | ¥0.05 | ¥0.60 | ¥0.08 | **¥0.73** |
| 600 页/30 章 | ¥0.08 | ¥0.90 | ¥0.12 | **¥1.10** |

> 按 DeepSeek-V3 输入 ¥1/百万 token、输出 ¥4/百万 token 估算；实际可能因章节长度和内容密度浮动。

## 四、坏情况处理

### 4.1 能处理的

| 问题 | 处理方式 |
|------|---------|
| PDF 是扫描版（无文字层） | 检测后提示"扫描版 PDF 无法提取文字，请使用 EPUB" |
| 章节识别失败 | 降级为均匀分块（每 5000 字一块），加注"自动分块" |
| 个别章节 AI 调用失败 | 重试 2 次，仍失败则跳过该章，最终报告标注 |
| 书内容太差/太水 | 阶段 1 骨架提取后让 AI 评估"可蒸馏度"，低于阈值则警告用户 |

### 4.2 不处理的（明确拒绝）

| 场景 | 原因 |
|------|------|
| 纯图片 PDF（漫画、画册） | 需要 OCR + Vision API，成本不可控 |
| 小说/诗歌 | 蒸馏不出方法论，不适合 |
| 单文件超大（>100MB） | 超时风险，拒绝上传 |
| 加密/DRM EPUB | 无法解析 |

## 五、Web 应用设计

### 5.1 页面结构

单页面应用，三个区域从上到下：

**区域 1 — 上传**
- 拖拽上传框（支持 PDF/EPUB/TXT，最大 50MB）
- 上传后显示：文件名、大小、页数/字数估算、格式图标

**区域 2 — 配置**
- 书名（自动从文件中提取，可手动修改）
- 输出语言（中文/English）
- 蒸馏深度：快速（仅骨架 + 章节索引）/ 标准（含每章详细提炼）/ 深度（附加术语表 + 速查表 + 习题）
- 几个复选框：术语表、速查表、章节习题

**区域 3 — 处理与结果**
- 点击"开始蒸馏"后显示进度（阶段 1 → 阶段 2[n/m] → 阶段 3 → 打包）
- 完成后显示：总 token 消耗、耗时、成本
- 大号"下载"按钮 + 文件结构预览

### 5.2 状态流转

```
[上传] → [配置] → [处理中] → [完成] → [下载]
                            ↘ [失败] → 显示错误 + 重试
```

### 5.3 处理进度设计（SSE 推送）

```
🔄 阶段 0：正在解析 PDF... ✓ (3.2s, 187K tokens)
🔄 阶段 1：正在提取全书骨架... ✓ (8.5s)
🔄 阶段 2：正在蒸馏章节 (15/20)... 
🔄 阶段 3：正在整合知识... 
🔄 阶段 4：正在打包... ✓
✅ 完成！总耗时 127s · API 成本 ¥0.68 · 总 token 82K
```

## 六、技术栈

| 层 | 技术 | 理由 |
|----|------|------|
| 后端框架 | **FastAPI** | Python 生态好，异步原生支持，文件上传方便 |
| 前端 | **单页 HTML + Vue 3 CDN** | 不需要构建工具，一个文件搞定，部署简单 |
| AI API | **DeepSeek API**（兼容 OpenAI SDK） | 性价比最高，代码零改动可切到其他模型 |
| PDF 解析 | **PyMuPDF (fitz)** | 速度快，中文支持好，自动识别章节 |
| EPUB 解析 | **ebooklib + BeautifulSoup4** | 标准方案 |
| 部署 | **Docker 单容器** | 迁移方便，环境一致 |
| 进度推送 | **SSE (Server-Sent Events)** | 比 WebSocket 简单，单向推送就够 |

## 九、关键设计决策记录

1. **不用 RAG/向量数据库** — 蒸馏场景需要理解和重构，不是检索。Map-Reduce 更合适。
2. **不用 OpenAI/Claude API** — DeepSeek 性价比碾压，且蒸馏任务对推理能力要求不高。
3. **不做用户系统** — MVP 阶段就是单机工具，不需要登录/注册/历史记录。
4. **两层输出** — the-knowledge-guy 验证过的方案，token 效率提升 24-51 倍。
5. **章节并行** — 阶段 2 是最大成本源，并行化让处理时间不随章节数线性增长。
