# LLM 重复循环：根因、诊断与工程化防御

> 记录于 2026-06-11，基于《刻意练习》蒸馏实战中发现的 DeepSeek 字符级重复问题。

---

## 现象

蒸馏《刻意练习》时，9 章中有 2 章出现严重的字符级重复：

| 章节 | 问题 | 输出长度 | 有意义行数 |
|------|------|----------|-----------|
| ch01 有目的的练习 | "史蒂夫·**夫夫夫夫夫夫夫夫夫夫**..." | 4218 chars | ~5 行 |
| ch04 黄金标准 | "它要求**你你你你你你你你你你**..." | 4112 chars | ~3 行 |
| 其余 7 章 | 正常 | 2000-4000 chars | 30-57 行 |

这不是"输出质量差"，而是模型进入了一个无法自愈的正反馈状态。

---

## 根因：曝光偏差（Exposure Bias）

### 训练 vs 推理的差异

LLM 训练使用 **Teacher Forcing**——每一步都看到人类写的"正确答案"作为上文：

```
训练时：  人类写的上文 → 预测下一个 token → 对比人类写的下一个 token → 更新权重
```

但推理时，模型看到的是**自己刚生成的 token**：

```
推理时：  自己生成的上文 → 预测下一个 token → 采样 → 这个 token 进入上文 → 预测再下一个
```

这个差异叫**曝光偏差**。训练时模型从未暴露在自己的错误中，所以它不知道如何从"自己造成的偏差"中恢复。

### 重复如何形成正反馈

假设某一步的 token 概率分布：

```
"夫": 8%
"的": 5%
"了": 4%
"这": 3%
... 其余都 < 3%
```

temperature=1.0 意味着严格按这个分布采样。假设"夫"被采样到（8% 的概率，不算低），它出现在上文里。下一步：

```
上文 = "...史蒂夫·夫"
"夫": 15%  ← 因为上文以"夫"结尾，"夫"后面跟"夫"的 bigram 概率上升
"的": 4%
...
```

一旦"夫夫"出现，它下一步的概率更高：

```
上文 = "...史蒂夫·夫夫"
"夫": 25%
...
```

这就是**正反馈循环**：重复 → 概率上升 → 更重复 → 概率更高 → 锁定。

整个过程不是一个"bug"，而是自回归生成在缺乏约束时的**必然数学行为**。

---

## 为什么 temperature=1.0 特别危险

temperature 控制采样的"锐度"：

| temperature | 行为 | 重复风险 |
|-------------|------|---------|
| 0.0 | 总是选概率最高的 token（贪婪解码） | 不重复单个字，但可能重复整个段落 |
| 0.3-0.7 | 概率分布适度平滑 | 低 |
| 1.0 | 严格按原始概率分布采样 | **最高** — 没有任何安全边际 |
| > 1.0 | 概率分布被拉平，更多低概率 token 被选中 | 重复风险下降，但输出变得随机/不连贯 |

temperature=1.0 理论上是"最像人的"，但代价是**零安全边际**。一旦某个 token 的重复概率开始上升，模型完全没有内在机制来对抗。

---

## 为什么 Claude 不这样，DeepSeek 会这样

**Claude** 的 RLHF/RLHA 训练流程中，抗重复行为已经被内化到模型权重里。训练数据中包含了大量重复文本的负面反馈，模型学会了即使在 temperature=1.0 下也能自我规避重复。这是**训练阶段的对齐**。

**DeepSeek** 的训练目标更偏 raw capability（推理能力、知识覆盖），没有把抗重复作为独立的安全目标来优化。这是设计选择，不是缺陷——它假设调用者会自己加参数控制。DeepSeek 的定位更接近"高性能引擎"，而不是"自带安全驾驶的整车"。

所以在 DeepSeek 上，你需要**手工搭建 Claude 已经内置的那层安全机制**。

---

## 三层防御体系

### 第 1 层：API 参数（概率层面）

DeepSeek API 支持两个关键参数：

| 参数 | 范围 | 作用机制 | 对重复的影响 |
|------|------|---------|------------|
| `frequency_penalty` | -2.0 ~ 2.0 | token 出现次数越多，概率被压制越多 | 直接打断字符级/词组级重复 |
| `presence_penalty` | -2.0 ~ 2.0 | token 只要出现过一次，概率就降低 | 鼓励引入新话题，防止原地兜圈子 |

两者都是**在采样前修改 logits**，不是事后过滤。所以它们能在重复还没开始时就阻止。

**参数选择逻辑**：

- `frequency_penalty=0.5`：一个字符连续出现 ~10 次后，它的概率会被压到原来的 ~1/3。足以打断"夫夫夫夫"这种字面重复，但虚词"的""是""了"本来就高频，需要重复很多次才会被明显压制——所以不会影响语言自然度。
- `presence_penalty=0.2`：轻微鼓励模型不要重复已经提过的概念/措辞。对章节蒸馏特别有用——防止每个框架都用"通过...可以实现..."这种套话。

**为什么不是更高的值**：`frequency_penalty=2.0` 会让模型完全不敢复用任何 token，输出变得生硬、不自然，甚至语法错误。

### 第 2 层：n-gram 检测（确定层面）

第 1 层是概率性的——降低重复的概率，但不**保证**。极端情况下（上下文已经严重偏向重复），penalty 可能不够。

第 2 层是**确定性的**——正则表达式不会漏：

```python
# 字符级：同一字符连续 10 次以上
r'(.)\1{10,}'

# 2-gram：同一对字符连续 5 次以上
r'(.{2})\1{5,}'

# 3-gram：同一组三字符连续 3 次以上
r'(.{3})\1{3,}'
```

成本为零（纯 CPU 计算），作为安全网兜底。检测到就抛异常，触发重试。

### 第 3 层：渐进重试（策略层面）

每次重试时**动态加大 penalty**：

| 尝试 | frequency_penalty | presence_penalty | 策略意图 |
|------|-------------------|------------------|---------|
| 第 1 次 | 0.5 | 0.2 | 正常参数，大概率成功 |
| 第 2 次 | 0.8 | 0.3 | 加大惩罚，应对边缘 case |
| 第 3 次 | 1.1 | 0.4 | 强力压制，牺牲一点自然度换确定性 |

这样确保不会用同样的参数撞两次墙。如果 3 次都失败（极少见），返回失败标记，整章标记为"蒸馏失败"——比静默保存 4000 字的"夫夫夫夫"好得多。

## 这不是幻觉

总结一个关键认知区分：

| 问题 | 定义 | 解决方向 |
|------|------|---------|
| **幻觉（Hallucination）** | 模型输出了与事实不符的内容 | RAG、grounding、fact-checking |
| **重复循环（Repetition Loop）** | 模型在自回归过程中进入正反馈锁定 | frequency_penalty、n-gram 检测、重试 |
| **退化（Degeneration）** | 输出质量逐渐下降，趋向无意义 | temperature 调优、nucleus sampling (top_p) |

这三者常被混淆，但根因和解决方案完全不同。我们遇到的是**重复循环，不是幻觉**。

---

## 适用范围

这套方案适用于：

- **DeepSeek 系列模型**（V3、R1 等）：训练目标偏 raw capability，需要外部 anti-repetition
- **其他未做 RLHF 的模型**：类似的 API 参数通常可用
- **temperature >= 0.8 的场景**：高温更需这层防护
- **长文本生成**：token 越多，进入重复循环的概率越高

不适用于：

- Claude API：已经有内化的抗重复机制，加 penalty 反而可能导致输出生硬
- temperature < 0.3：低温度下重复模式不同（整段重复而非字符重复），可能需要 min_p 或 top_p 采样
